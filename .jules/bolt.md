## 2024-03-28 - [Locator-X Levenshtein Distance Optimization]
**Learning:** The auto-locator healing algorithm in `locator-generator.js` relies heavily on fuzzy matching using the `levenshtein` distance function. The original implementation used an unoptimized 2D matrix allocation strategy (`O(N*M)` memory), which scaled poorly and ran slowly when evaluating many matching elements (e.g., tags, IDs, classes) during rapid DOM scanning. Replacing this with a 1D array approach that leverages prefix/suffix trimming and `charCodeAt` reduced memory overhead to `O(min(N, M))` and improved execution speed by roughly 18x (from ~565ms to ~30ms for 10000 iterations).
**Action:** Always check the implementation of string distance algorithms (like Levenshtein) in high-frequency validation loops. Optimize by using 1D arrays for dynamic programming and stripping common prefixes/suffixes before full evaluation.

## 2024-03-29 - [Locator-X validateLocator Array Allocation Optimization]
**Learning:** In performance-sensitive DOM operations like the auto-healing validation logic in `locator-generator.js`'s `validateLocator` method, converting large `NodeList` results (via `querySelectorAllDeep`) into an array using `Array.from(elements).filter(...)` introduces massive and unnecessary `O(N)` memory allocation overhead and prevents early loop termination. When testing against DOMs containing many nodes, scanning the entire node list completely negates limits like `maxCap`, taking 180ms+ for 100,000 nodes.
**Action:** Always prefer iterating directly over a `NodeList` (or similar iterable structures) using a standard `for` loop with a `break` condition when applying threshold limits. This approach reduces execution time by roughly 70% and drastically lowers memory pressure since only valid matches are pushed into a tiny fixed-limit array.
