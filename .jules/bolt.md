## 2024-03-28 - [Locator-X Levenshtein Distance Optimization]
**Learning:** The auto-locator healing algorithm in `locator-generator.js` relies heavily on fuzzy matching using the `levenshtein` distance function. The original implementation used an unoptimized 2D matrix allocation strategy (`O(N*M)` memory), which scaled poorly and ran slowly when evaluating many matching elements (e.g., tags, IDs, classes) during rapid DOM scanning. Replacing this with a 1D array approach that leverages prefix/suffix trimming and `charCodeAt` reduced memory overhead to `O(min(N, M))` and improved execution speed by roughly 18x (from ~565ms to ~30ms for 10000 iterations).
**Action:** Always check the implementation of string distance algorithms (like Levenshtein) in high-frequency validation loops. Optimize by using 1D arrays for dynamic programming and stripping common prefixes/suffixes before full evaluation.

## 2024-05-18 - [Locator-X Fuzzy Matching Memory Optimization]
**Learning:** In `src/services/locator-generator.js`, the `validateLocator` method was using `Array.from(elements).filter(...)` to find fuzzy matches for broken locators. This pattern is problematic because it allocates a new array for all elements returned by `querySelectorAllDeep` (which could be thousands), runs the expensive Levenshtein distance calculation for *every* element, and only checks the match count limit `finalLimit` afterward. This eager evaluation caused significant memory spikes and wasted CPU cycles.
**Action:** For performance-sensitive DOM operations, especially those involving expensive string distance calculations over large element collections, avoid `Array.from().filter()`. Use native `for` loops to iterate over `NodeList` collections and implement early exit conditions (e.g., `if (count >= finalLimit) break;`) to stop processing as soon as the required number of matches is found.
